## 1. Резюме

Проект `rusa` — это Python-утилита для автоматического закадрового перевода видео по субтитрам с использованием TTS (Microsoft Edge TTS или произвольная команда через `--tts-cmd`). Код разделён на 7 модулей, имеет тесты, CI, кэширование TTS/WAV и управление субтитрами в выходном контейнере.

**Общая оценка:** проект в стабильном состоянии, но содержит ряд латентных багов, проблем с документацией и одну тестовую регрессию, которая падает в офлайн-окружении. Основная цель последнего коммита (`0185b91`) — исправить генерацию выходного имени файла для `--tts-cmd` — достигнута, но связанные риски (shell-инъекции, некорректные суффиксы языка) остаются.

## 2. Контекст коммита `0185b91`

**Изменения:**
- Файл: `rusa_shared.py`, строка 579
- `base = exe.rsplit(".exe", 1)[0]` → `base = os.path.basename(exe).rsplit(".exe", 1)[0]`

**Проблема, которую исправили:**
При использовании `--tts-cmd '/home/user/bin/tts_silero.py {in} {out} {voice}'` в `_display_name` попадал полный путь, и имя выходного файла содержало слэши: `movie_/home/user/bin/tts_silero.py_baya.mkv`.

**Оценка исправления:**
- Корректно и минимально.
- Убирает только проблему пути; не затрагивает другие риски `CustomCmdBackend` (shell=True, неэкранированные плейсхолдеры, выбор голоса).
- Регрессионных тестов на этот сценарий нет.

### Единственный упавший тест

**`tests/test_regression.py::test_voice_resolution_with_external_srt`**

```python
detected = rusa.detect_language_from_srt(srt_path)
assert detected is not None, "Should detect language from ru_subtitles.srt"
```

**Причина падения:**
1. Фикстура называется `ru_subtitles.srt`, а не `*.ru.srt`.
2. Регулярка `re.match(r".*\.([a-z]{2,7})\.srt$", name)` ожидает паттерн `.<код>.srt` (например, `movie.ru.srt`). Для `ru_subtitles.srt` совпадения нет.
3. Фallback — анализ содержимого через `langdetect`. В окружении аудита `langdetect` не установлен (`HAS_LANGDETECT = False`), поэтому функция возвращает `None`.

**Вердикт:** это **баг в тесте**, а не в production-коде. Тест либо должен использовать фикстуру `en.ru_subtitles.srt` (которая как раз имеет вид `*.ru.srt`), либо не должен полагаться на `langdetect` в офлайн-режиме. Рекомендуется заменить `ru_subtitles.srt` на `en.ru_subtitles.srt` или явно проверять `HAS_LANGDETECT`.

### Другие замечания по тестам
- `edge-tts` не установлен в окружении аудита; 4 live-теста корректно пропущены.
- `ffmpeg`/`ffprobe` пришлось подкладывать статическим бинарником; CI устанавливает их через `apt`.

## 4. Безопасность

### 4.1. `shell=True` в `CustomCmdBackend.generate` (средний/высокий риск)

**Файл:** `rusa_shared.py:624`

```python
cmd = (cls._cmd_template
       .replace("{in}", in_file)
       .replace("{out}", out)
       .replace("{voice}", voice))
rc = subprocess.run(cmd, shell=True, capture_output=True, timeout=180)
```

**Проблемы:**
- Пользовательский шаблон передаётся в shell без предварительной валидации.
- Плейсхолдеры `{in}`, `{out}`, `{voice}` подставляются простой `replace()`, без экранирования кавычек/пробелов/спецсимволов.
- Если пользователь использует `--voice` со значением, содержащим `;`, `|`, `$`, `\``, это может привести к выполнению произвольных команд.
- Даже без злого умысла: путь к временному файлу может содержать пробелы или апострофы, что сломает команду.

**Рекомендации:**
1. Валидация шаблона: разрешить только один раз `{in}`, `{out}`, `{voice}`.
2. Экранировать подставляемые значения через `shlex.quote`.
3. Альтернативно: разобрать шаблон на список аргументов и запускать с `shell=False`.
4. Добавить unit-тест на `--tts-cmd` с путями, содержащими пробелы и кавычки.

### 4.2. Путь к выходному файлу формируется из пользовательских данных (низкий риск)

**Файл:** `rusa.py:143-149`

```python
output = os.path.join(
    os.path.dirname(video),
    f"{base}_{getattr(backend_cls, '_display_name', backend_cls.name)}_{lang_suffix}{ext}",
)
```

После коммита `0185b91` путь очищается `os.path.basename`, но `lang_suffix` всё ещё берётся из `backend_cls.lang_from_voice(voice)` (для `CustomCmdBackend` это полное имя голоса). Если голос содержит `/` или другие запрещённые символы, имя файла может быть некорректным. Рекомендуется санитизировать `lang_suffix` аналогично `_display_name`.

### 4.3. `tmpdir` создаётся через `tempfile.mkdtemp` (хорошо)

Использование стандартной библиотеки — правильный подход. Временные файлы удаляются в `finally`, если не включён `--keep-temp`.

### 4.4. Кэш по SHA-256 от голоса + текста (хорошо)

Ключи кэша детерминированы и не зависят от путей. Есть защита от перезаписи существующих файлов.

### 4.5. `subprocess.run` в остальных местах — без `shell=True`

Большинство вызовов `ffmpeg`/`ffprobe`/`edge-tts` используют список аргументов. Это безопасно.

## 5. Качество кода и баги

### 5.1. Фильтрация голосов в `list_voices` сломана при `--lang`

**Файл:** `rusa_cli.py:115-118`

```python
if lang and f"-{lang}-" not in line.split(":")[-1].strip():
    continue
```

**Проблема:** при `--voice --lang ru` фильтр ищет подстроку `-ru-` в строке вида `Name: ru-RU-SvetlanaNeural`. Строка `ru-RU-SvetlanaNeural` содержит `ru-`, но не `-ru-`, поэтому все русские голоса будут отфильтрованы.

**Рекомендация:**
```python
voice_part = line.split(":")[-1].strip().lower().replace("_", "-")
if lang and not voice_part.startswith(f"{lang}-"):
    continue
```

### 5.2. Некорректные языковые алиасы для неподдерживаемых языков

**Файл:** `rusa_shared.py`, `LANG_ALIAS`

Примеры:
- `"norwegian": "no"` — но `LANG_VOICE_MAP` и `LANG_FFPROBE_MAP` используют ключ `nb`, а не `no`. Результат: `--lang norwegian` приводит к ошибке "Язык 'no' не поддерживается".
- `"ukrainian": "uk"`, `"hindi": "hi"`, `"indonesian": "id"`, `"thai": "th"`, `"vietnamese": "vi"`, `"greek": "el"`, `"romanian": "ro"`, `"croatian": "hr"`, `"serbian": "sr"`, `"bulgarian": "bg"`, `"malay": "ms"`, `"slovak": "sk"` — все эти алиасы ведут на ISO-коды, отсутствующие в `LANG_VOICE_MAP`/`LANG_FFPROBE_MAP`, и приводят к той же ошибке.

**Рекомендация:** либо добавить голоса для этих языков, либо убрать алиасы, либо выдавать понятное сообщение "Язык определён, но нет встроенного голоса; укажите --voice".

### 5.3. Автовыбор голоса для `--tts-cmd` некорректен

**Файл:** `rusa.py:121-127`

```python
if args.voice is None and args.lang is None:
    detected = detect_language_from_srt(subs_path)
    if detected:
        lang_code = voice_to_lang_code(detected)
        voice = backend_cls.get_default_voice(lang_code) or detected
```

**Проблема:** для `CustomCmdBackend` `get_default_voice` всегда возвращает `None`, поэтому `voice` остаётся значением, подходящим для Edge TTS (например, `ru-RU-SvetlanaNeural`), а не для пользовательского движка. Это может привести к тому, что `espeak-ng` или `RHVoice` получат невалидный голос.

**Рекомендация:** для `CustomCmdBackend` требовать явный `--voice` или использовать `--lang` для выбора языка субтитров, а голос оставлять на усмотрение пользователя.

### 5.4. `voice_to_lang_code` для коротких голосов даёт мусор

**Файл:** `rusa_shared.py:174`

```python
def voice_to_lang_code(voice: str) -> str:
    return voice[:2].lower()
```

Для голоса `baya` (Silero) возвращается `ba`, для `elena` (RHVoice) — `el`. Это используется в имени выходного файла и в `voiceover_lang` метаданных ffmpeg. Некорректный код языка (`ba`, `el`) может запутать плееры.

**Рекомендация:** для `CustomCmdBackend` использовать `--lang` как единственный источник языка, а не пытаться извлечь код из голоса.

### 5.5. `voiceover_lang` метаданные могут быть невалидны

**Файл:** `rusa.py:170`

```python
voiceover_lang = lang_code_to_ffprobe_codes(target_lang)[0] if target_lang else "rus"
```

Если `target_lang` — это код вроде `baya` (неизвестный), `lang_code_to_ffprobe_codes` возвращает `["baya"]`, и ffmpeg получает `-metadata:s:a:1 language=baya`, что не является стандартным ISO-кодом.

**Рекомендация:** если `target_lang` неизвестен, fallback на `und` или требовать `--lang`.

### 5.6. `list_voices` не поддерживает `CustomCmdBackend`

`rusa_cli.list_voices` всегда выводит только `edge-tts`. При `--tts-cmd` и `--voice` список голосов не имеет смысла, но `main()` всё равно вызывает `list_voices(args.lang)`.

**Рекомендация:** при `--tts-cmd` выводить сообщение "Произвольная TTS-команда: список голосов недоступен".

### 5.7. `_split_text` может разрезать на границе, не сохраняя пробел

**Файл:** `rusa_tts.py:21-39`

```python
for match in re.finditer(r"[.!?…]", chunk):
    split_at = match.end()
```

Алгоритм берёт **последнее** совпадение знака препинания в пределах `max_chars`. Если в предложении есть несколько точек (например, сокращения), это может разрезать текст в неожиданном месте. Кроме того, при разбиении по пробелу `chunk.rfind(" ")` может вернуть `-1`, и тогда `split_at = max_chars`, что обрезает слово.

**Рекомендация:** использовать более умное разбиение по предложениям с учётом пробелов после знаков препинания.

### 5.8. `print_cache_stats` / `clear_cache` используют `sys.exit(0)`

**Файл:** `rusa_shared.py:400-420`

Это допустимо, но затрудняет тестирование и повторное использование в API. Тесты используют `pytest.raises(SystemExit)`, что нормально, но лучше бы выделить логику отдельно от выхода.

### 5.9. `step_mix_output` содержит мёртвый код для default output

`output` всегда устанавливается в `main()` (строки 143–149), поэтому ветки `if not output` в `step_mix_output` (строки 282 и 298) никогда не выполняются.

**Рекомендация:** удалить дублирующий fallback или сделать `output` обязательным параметром.

### 5.10. `copy_into_cache` не проверяет, что файл действительно в кэше

```python
cached_dir = os.path.dirname(cache_path) if cache_path else None
if cached_dir and os.path.isdir(cached_dir):
    _evict_oldest(cached_dir)
```

`cache_path` может быть `None` (кэш отключён), тогда eviction пропускается. Это корректно.

### 5.11. `cache_bucket_stats` обходит дерево рекурсивно, а eviction — только один уровень

Несоответствие логики: если в кэше когда-либо появятся поддиректории, статистика и eviction будут считать по-разному. В текущей реализации поддиректорий нет, но это потенциальная ловушка.

### 5.12. Нет проверки на перезапись выходного файла

`rusa` перезаписывает `output` с флагом `-y` ffmpeg без предупреждения. Это может быть неожиданно для пользователя.

**Рекомендация:** добавить подтверждение или флаг `--overwrite` (по умолчанию `True` для CLI, но можно документировать).

## 8. Производительность

### 8.1. Кэширование TTS и WAV

- TTS-кэш: SHA-256 от `(backend, voice, text)`.
- WAV-кэш: SHA-256 от `(mp3_sha256, speed, filter_version)`.
- LRU-eviction по `RUSA_CACHE_MAX_SIZE` (по умолчанию 2 GiB).
- Это хорошо и снижает повторные вызовы сети/ffmpeg.

### 8.2. `_check_ffmpeg_codec` закэширован через `lru_cache(maxsize=1)`

Хорошо. Устраняет множественные вызовы `ffmpeg -encoders`.

### 8.3. Параллельная конвертация WAV

`ThreadPoolExecutor` с `threads` воркерами. Это корректно для I/O-bound задач с ffmpeg.

### 8.4. Параллельная генерация TTS

`ThreadPoolExecutor` с `threads` воркерами. Для edge-tts это сетевые вызовы — потоки оправданы. Однако при большом числе потоков можно получить rate-limiting со стороны сервера Microsoft.

### 8.5. Сборка voiceover — один write pass

`step_assemble` открывает файл один раз в режиме `wb` и пишет последовательно. Это улучшение по сравнению с предыдущими версиями. WAV-заголовок корректируется в конце. Проверено тестами.

### 8.6. `EdgeTtsBackend.validate_voice` вызывает `list-voices` каждый раз

При указании `--voice` происходит дополнительный сетевой/субпроцессный вызов `edge_tts --list-voices`. Это замедляет запуск. Можно кэшировать результат на уровне сессии.

## 9. Архитектура

### 9.1. Модульное разделение

| Модуль | Ответственность | Состояние |
|--------|-----------------|-----------|
| `rusa.py` | Тонкий entrypoint + `main()` | Хорошо |
| `rusa_cli.py` | CLI парсер + `list_voices` | Есть баг с фильтром |
| `rusa_subtitle.py` | SRT: extract, detect, parse, sync, merge | Хорошо |
| `rusa_tts.py` | Генерация TTS | Хорошо, но split-text можно улучшить |
| `rusa_audio.py` | MP3 → WAV + assembly | Хорошо |
| `rusa_mux.py` | Микс + mux + кодеки | Хорошо, но есть мёртвый код |
| `rusa_shared.py` | Константы, кэш, бэкенды, терминал | Хорошо, но есть баги алиасов и shell-риск |

### 9.2. TTS Backend Abstraction

`TtsBackend`, `EdgeTtsBackend`, `CustomCmdBackend` — правильная абстракция. `BACKEND_REGISTRY` позволяет добавлять бэкенды. Однако `CustomCmdBackend` имеет особенности, которые ломают логику `main()` (см. разделы 5.3, 5.4).

### 9.3. `--tts-backend` удалён

Единый встроенный бэкенд — `edge`. Пользовательские движки — через `--tts-cmd`. Это упрощает код, но README.en.md ещё не отражено.
